# ARIA ver.2

ARIA ver.2는 특허 청구항과 인용발명 PDF를 비교하여 신규성·진보성 판단 보고서를 생성하는 로컬 AI 보조 프로그램입니다. 기존 ARIA와 동시에 실행할 수 있도록 ver.2 전용 포트와 실행 스크립트를 사용합니다.

## 비교 입력 전략

설정의 **인용발명 비교 방식**에서 두 모드 중 하나를 선택할 수 있습니다.

- **문헌별 순차 대비(기본값)**: 청구항 하나를 인용발명마다 별도의 LLM 호출로 대비합니다.
- **전체 통합 대비(LLM 1회)**: 청구항 하나와 모든 인용발명을 하나의 프롬프트로 묶어 한 번의 LLM 호출로 대비합니다.
- 두 모드 모두 입력 텍스트가 한도 이내이면 인용발명 전문을 그대로 사용합니다.
- 전체 통합 대비에서 합산 텍스트가 입력 한도를 넘으면 모든 문헌을 유지하면서 문헌별 관련 문단을 RAG 또는 키워드 기반으로 압축합니다.
- RAG는 BGE-M3 dense 검색과 SQLite BM25 결과를 결합하고, 선택적으로 `bge-reranker-v2-m3`로 후보 문단을 재정렬합니다. Reranker 실패 시 기존 RRF 순서로 자동 대체되며 추가 LLM 호출은 발생하지 않습니다.
- 비교 방식 또는 판정·보고서 정책 버전이 바뀌면 이전 구성대비·보고서 캐시는 자동으로 갱신됩니다.
- 종속항 일괄 보고서는 각 종속항의 추가 구성에 대해 RAG로 후보 인용발명을 먼저 고른 뒤 후보 문헌을 정밀 비교합니다. 후보가 없거나 불확실하면 전체 문헌 비교로 되돌아갑니다.

## 독립항 인용발명 결합 정책

- 독립항의 진보성 판단은 원칙적으로 주인용발명 1개와 보조인용발명 1개, 최대 2개를 결합합니다.
- 주인용발명의 미커버 구성을 실제 발췌 근거로 보완하는 문헌만 보조인용발명으로 채택하며, 보완 문헌이 없으면 보완 인용발명 없음으로 표시합니다.
- 남은 구성이 CPU, 바퀴, 일반 제어부처럼 낮은 중요도의 독립적인 통상 구성이고 특수한 수치·조건·상호작용 제한이 없으면 주지관용 검토 대상으로 분리합니다.
- 해당 통상 구성을 강하게 개시한 문헌이 있으면 `주지관용 구성의 문헌 근거` 역할로 표시하고, 핵심 기술사상의 보완 근거로 확대하지 않습니다.
- 핵심 구성에 이미 인용발명 1과 2가 필요한 경우에만, 남은 통상 구성의 명시 근거로 인용발명 3을 예외적으로 허용합니다.
- 예외적 인용발명 3은 대상 통상 구성에 `동일` 또는 `실질적 동일` 판정과 원문 발췌가 있을 때만 채택합니다.
- 적절한 문헌 근거가 없으면 인용발명 1 또는 인용발명 1·2와 주지관용 검토를 병기하되, 문헌 근거 없이 주지관용이라고 단정하지 않고 추가 입증 필요 여부를 표시합니다.

## 종속항 인용발명 결합 정책

- 종속항은 직접 인용하는 부모항의 인용발명 조합을 전부 상속합니다.
- 해당 종속항의 추가 구성에 남은 공백이 있으면, 그 공백 전부를 대응시키는 인용발명 1개만 새로 추가할 수 있습니다.
- 하나의 종속항을 거절하기 위해 새 인용발명 2개 이상을 추가로 결합하지 않습니다. 서로 다른 두 문헌이 있어야 추가 구성이 모두 채워지는 경우에는 해당 조합만으로 거절 근거가 완성되지 않은 것으로 처리합니다.
- 따라서 독립항이 인용발명 1·2로 검토되고 청구항 2가 인용발명 3을 추가하면 청구항 2의 최종 조합은 1·2·3입니다. 청구항 2를 인용하는 청구항 3은 여기에 인용발명 4를 하나 더 추가할 수 있고, 독립항을 직접 인용하는 별도 청구항은 1·2에서 다시 시작해 자기 추가 구성에 대응하는 문헌 1개만 더합니다.

## 주요 기능

- 인용발명 PDF 최대 7개 업로드 및 텍스트·문단·페이지 정보 추출
- 사용자가 붙여넣은 청구항을 구성요소 단위로 분해
- 구성요소별 인용발명 대비 결과와 최종 거절이유 보고서 생성
- 독립항·종속항 흐름을 반영한 일괄 보고서 생성
- Claude CLI, OpenAI Codex CLI 또는 AGY CLI 기반 LLM 실행
- 공백 구성요소 키워드 기반 보완문헌 검색 전략 및 보고서 Q&A 지원
- PyMuPDF 우선 추출, OpenDataLoader PDF fallback, SQLite/Qdrant 기반 RAG 저장 구조

## 빠른 시작

PowerShell에서 다음 명령을 실행합니다.

```powershell
.\start.ps1
```

스크립트는 백엔드 의존성을 설치하고 FastAPI 서버와 Vite 프론트엔드를 실행합니다.

## 개별 실행

### Backend

```powershell
pip install -r backend/requirements.txt
python -m uvicorn backend.main:app --host 0.0.0.0 --port 8200 --reload
```

### Frontend

```powershell
cd frontend
npm install
npm run dev
```

## 기본 사용 흐름

1. 인용발명 PDF를 업로드하고 준비 작업을 실행합니다.
2. 청구항 번호와 청구항 전문을 입력합니다.
   - 복수 청구항 헤더는 `청구항 N`, `제N항`, `【청구항 N】`, `[CLAIM N]` 형식을 인식합니다.
   - 종속항이 참조하는 부모항이 입력 또는 이전 컨텍스트에 없으면 부모항의 실체를 추정하지 않고, `에 있어서` 뒤에 직접 기재된 기술 특징만 인용발명과 대비합니다.
3. 필요하면 설정에서 `문헌별 순차 대비` 또는 `전체 통합 대비(LLM 1회)`를 선택합니다.
4. 보고서를 생성하여 Phase 1 구성요소 대비와 Phase 2 최종 보고서를 확인합니다.
5. 필요한 경우 키워드 보완문헌 검색 또는 Q&A 패널로 판단 근거를 보강합니다.
